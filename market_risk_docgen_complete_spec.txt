
MARKET RISK DOCUMENTATION GRAPH GENERATOR
COMPLETE IMPLEMENTATION SPECIFICATION (V1)

1. OBJECTIVE

Build a Python application that generates a fully static HTML documentation site for market-risk systems.

The documentation is represented as a graph composed of Data nodes and Process nodes.

The generated output must:
- Work without a backend
- Work without a database
- Work without internet access
- Support local MathJax rendering
- Support navigation through hyperlinks
- Support centralized notation management
- Support centralized references
- Support multiple documentation profiles

2. CORE CONCEPTS

2.1 Node Types

Only two node types exist:

DATA NODE
Represents a business or technical data item.

Examples:
- IRCurves
- IRReturns
- RiskFactors
- VaRResult

PROCESS NODE
Represents a transformation, model, algorithm, workflow, or calculation.

Examples:
- CalculateIRReturns
- HistoricalVaR
- MonteCarloVaR

2.2 Graph Rules

Allowed:

Data -> Process
Process -> Data

Forbidden:

Data -> Data
Process -> Process

Every process:
- Has at least one input
- Has at least one output

Inputs and outputs must always be data nodes.

3. DOCUMENTATION PHILOSOPHY

The documentation answers:

“How is this data produced?”

Documentation is generated by traversing lineage relationships.

Example:

IRReturns
 <- CalculateIRReturns
 <- IRCurves

5. NODE STORAGE

Each node is a Markdown file.

Structure:

---
yaml metadata
---

markdown body

6. COMMON NODE FIELDS

Required:

id
title
node_type

Optional:

description
tags
owner
status

7. DATA NODE SCHEMA

Required:

id: string
title: string
node_type: data
role: source | derived

Optional:

notation

Example:

notation:
  fields:

    tenor:
      symbol: "T"
      label: "tenor"

    return_value:
      symbol: "\Delta r"
      label: "interest-rate return"

8. PROCESS NODE SCHEMA

Required:

id: string
title: string
node_type: process

inputs:
  - DataNodeId

outputs:
  - DataNodeId

Optional:

notation_aliases

Example:

notation_aliases:
  curves: IRCurves
  returns: IRReturns

9. NOTATION SYSTEM

Notation belongs exclusively to data nodes.

Process nodes may consume notation.

Example:

Data node:

notation:
  fields:
    tenor:
      symbol: "T"

Process node:

{{ returns.tenor.symbol }}

returns is defined through:

notation_aliases:
  returns: IRReturns

10. NOTATION EXPRESSIONS

Support:

{{ alias.field.symbol }}
{{ alias.field.label }}
{{ alias.field.description }}
{{ alias.field.unit }}

Example:

{{ returns.return_value.symbol }}

resolves to:

\Delta r

11. NOTATION RESOLUTION ALGORITHM

During rendering:

1. Parse notation aliases.
2. Resolve alias -> data node.
3. Resolve field.
4. Resolve property.
5. Replace placeholder.
6. Continue rendering.

12. LATEX SUPPORT

Notation replacement must occur before Markdown conversion.

Example:

Source:

\[
{{ returns.return_value.symbol }}
=
{{ curves.rate.symbol }}
\]

After substitution:

\[
\Delta r
=
r(t,T)
\]

13. MATHJAX

Use local MathJax only.

Never use a CDN.

HTML must contain:

<script defer src="assets/mathjax/tex-mml-chtml.js"></script>

Builder receives:

--mathjax-dir

and copies MathJax assets.

14. REFERENCE REGISTRY

Central file:

references.yml

Example:

references:

  basel_frtb:
    label: "BCBS FRTB"
    title: "Minimum capital requirements for market risk"

15. REFERENCE USAGE

Inside Markdown:

{{ ref.basel_frtb }}

Render:

<a href="references.html#basel_frtb">
[BCBS FRTB]
</a>

16. REFERENCES PAGE

Generate:

references.html

Every reference receives a stable anchor.

17. PROFILES

Profiles define documentation choices.

Example:

id: regulatory_var

title: Regulatory VaR Documentation

root: IRReturns

producer_choices:

  IRReturns: CalculateIRReturns

18. PRODUCER SELECTION

Multiple producers may exist.

Example:

HistoricalVaR -> VaRResult
MonteCarloVaR -> VaRResult

Profile chooses one.

Only selected producer participates in traversal.

19. GRAPH MODEL

Build:

data_producers
data_consumers

20. TRAVERSAL

Start from profile root.

Traverse upstream.

Maintain visited set.

21. CYCLE HANDLING

If node already visited:

- Stop recursion
- Reuse existing page
- Emit hyperlink

Never recurse infinitely.

22. VALIDATION

Validate:

- Unique node ids
- Valid node types
- Valid roles
- Existing inputs
- Existing outputs
- Process inputs are data nodes
- Process outputs are data nodes
- Valid aliases
- Valid notation references
- Valid references
- Valid profile choices

Fail build on errors.
